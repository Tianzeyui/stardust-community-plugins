# 插件设计规范

## 目录结构

```
my-plugin/
  manifest.json          ← 插件声明（必需）
  index.tsx              ← React/TSX 入口（必需）
  package.json           ← npm 依赖声明（必需，即使为空）
  supabase/              ← 数据库 SQL（如有）
    └── YYYYMMDD_desc.sql
  docs/                  ← 文档（可选）
```

## manifest.json 规范

```json
{
  "id": "my-plugin",              // 唯一标识，全小写短横线
  "name": "我的插件",              // 显示名称
  "version": "1.0.0",            // 语义化版本
  "description": "一句话描述",     // 顶栏/列表中显示
  "icon": "Package",             // lucide-react 图标名
  "navOrder": 90,                // 侧边栏排序，越小越靠前
  "enabled": true,               // 安装后默认启用
  "systemHeader": false,         // false=自定义顶栏, true=系统渲染
  "permissions": ["ai", "files"],// 权限声明
  "requires": {                   // 依赖其他插件（可选）
    "plugins": ["other-plugin"]
  },
  "tools": []                     // AI 工具声明（可选）
}
```

## 设计参考

以下建议供参考，非强制要求。可根据实际需求灵活选择。

### 参考：日记插件的实现方式

日记插件（`plugins/diary/`）是符合宿主整体风格的一个参考实现：

1. **简洁入口** — `index.tsx` 仅做路由注册
   ```tsx
   export function register(ctx: any) {
     ctx.registerNav({ id: 'diary', label: '日记', icon: 'BookOpen', order: 60 })
     ctx.registerRoute('diary', () => import('@/components/diary/DiaryPage'))
   }
   ```

2. **业务逻辑独立** — 页面组件放在 `src/components/` 下，与插件入口分离

3. **自管理布局** — 设 `systemHeader: false`，页面自己渲染完整 UI

4. **SQL 随插件走** — 数据库 schema 放入 `supabase/` 目录

### 开发建议（可选）

- 推荐用宿主已有的 shadcn/ui 组件和 Tailwind CSS token，当然你也可以用自己的方案
- 异步操作建议处理 loading/error 状态
- 权限声明按实际需要，不多不少
